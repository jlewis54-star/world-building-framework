# Writing Standards

Clear, direct, data-driven communication for worldbuilding teams.

## Core principles

### 1. Be specific: use numbers, not adjectives

```
❌ "We saw significant improvement in the review cycle"
✅ "Asset review turnaround dropped from 12 days to 4 days (67% faster) in March"

❌ "The merch run had substantial cost overruns"
✅ "Unit cost rose $2.40 per SKU (18% over quote) because of foil stamp change"

❌ "Many partners missed the guideline"
✅ "3 of 11 vendors used the wrong typeface on packaging mockups"
```

### 2. Active voice: subject does the action

```
❌ "The manifesto was completed by the team on Friday"
✅ "The team completed the manifesto on Friday"

❌ "It has been determined that the palette conflict was..."
✅ "We traced the palette conflict to..."

❌ "Changes will be made to the channel playbook"
✅ "We will update the channel playbook"
```

### 3. Pyramid structure: most important first

```
❌ "We reviewed mood boards, polled three writers, checked competitor decks,
and after several sessions decided to lock the silhouette rule."

✅ "We lock the silhouette rule this week (decision). Rounded forms are
reserved for care objects only; institutions stay rectangular. Mood boards,
writer poll, and three packaging tests support the call."
```

### 4. Eliminate weasel words

```
❌ Words to remove: "very", "quite", "somewhat", "approximately", "generally",
   "typically", "usually", "often", "we believe", "it seems", "arguably"

✅ Replace with specifics:
   "very important" → "blocks sign-off"
   "approximately half" → "48%"
   "we believe this reads on-brand" → "Shipping Ledger friction check passed"
   "it seems likely" → "two test assets failed the grid check"
```

### 5. Short paragraphs and bullet points

- Paragraphs: 3-5 sentences maximum in prose
- Use bullets for 3+ parallel items
- Use numbered lists for sequential steps
- Never use tables for fewer than 3 rows of data

### 6. Precision in asks

Every communication that needs action must have:

- **Clear ask** (not implied)
- **Owner** named (not "the team")
- **Deadline** specified (not "soon" or "ASAP")

```
❌ "We need someone to review the lore section soon."
✅ "Morgan: review `12_lore_canon` timeline rows 4-7, comment by Friday EOD."
```

## Review checklist

- [ ] Lead sentence states the main point (not "In this document, we will...")
- [ ] All claims have data, paths, or examples
- [ ] No passive voice in key sentences
- [ ] All action items have owner + deadline
- [ ] No jargon without definition for the audience
- [ ] Length appropriate: email ≤200 words, 1-pager ≤500 words, doc ≤2000 words
- [ ] Subject/title describes what's in the document (not "Update" or "FYI")
- [ ] If target is `world/`: no scaffold sections leaked in

## Common rewrites

### Email subject lines

```
❌ "Update" → ✅ "Season 2 Pitch Deck: Lore Gap on Faction Names, Due Thursday"
❌ "FYI" → ✅ "Vendor Mockups Failed Grid Check: Three Fixes Before Print"
❌ "Quick question" → ✅ "Approval Needed: Public Positioning Paragraph for Partner PDF"
```

### Opening lines

```
❌ "I hope this email finds you well. I wanted to reach out regarding..."
✅ "Three launch assets fail the Shipping Ledger vocabulary check. Fixes inside."

❌ "As we discussed in our last meeting, there are several areas where..."
✅ "Two decisions from Tuesday's creative review need your call by EOW."
```

### Bullet points

```
❌ • The visual core has been updated
   • Testing was performed
   • It was sent to partners

✅ • Locked accent palette in `22_visual_core` (hex + Pantone)
   • Ran brand-friction on partner one-pager (5/5 pass)
   • Shipped PDF to vendors March 10
```
