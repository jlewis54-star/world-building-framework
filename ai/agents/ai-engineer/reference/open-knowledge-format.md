# Open Knowledge Format (OKF) v0.1

OKF is an open format for representing knowledge (the metadata, context, and curated insight around data and systems) as a directory of markdown files with YAML frontmatter. It is authored by people, generated by agents, and read by both. No schema registry, no central authority, no required tooling. If you can `cat` a file you can read it; if you can `git clone` a repo you can ship it.

Use this as the default structure whenever you produce a knowledge corpus: research folders, account bundles, reference libraries, project knowledge. It is minimally opinionated and standardizes only what is needed to make a corpus self-describing.

## When to apply OKF

- Building or enriching a folder of knowledge notes meant to be read by agents and humans.
- Structuring per-account, per-topic, or per-project research that other agents will traverse.
- Any corpus that should stay portable, diffable, and readable without bespoke tooling.

Do not use it to replace domain schemas (Avro, Protobuf, OpenAPI). OKF references those; it does not subsume them.

## Core model

- **Bundle**: a directory tree of markdown files. The unit of distribution (git repo, tarball, or a subdirectory of a larger repo).
- **Concept**: one markdown document. May describe a tangible asset (table, API) or an abstract idea (metric, playbook, account).
- **Concept ID**: the file path within the bundle minus the `.md`. `tables/users.md` has ID `tables/users`.
- **Frontmatter**: the YAML block at the top, delimited by `---`.
- **Body**: everything after the frontmatter.

Directory structure is domain-independent. Organize concepts however fits the knowledge.

## Reserved filenames

Only two, at any level. Never use them for concept documents.

| Filename | Purpose |
|----------|---------|
| `index.md` | Directory listing for progressive disclosure (see below) |
| `log.md` | Chronological update history |

Every other `.md` file is a concept document.

## Concept frontmatter

```yaml
---
type: <Type name>                  # REQUIRED, the only required field
title: <display name>              # recommended
description: <one-line summary>     # recommended
resource: <canonical URI>          # recommended when a physical asset exists
tags: [<tag>, <tag>]               # optional
timestamp: <ISO 8601 datetime>     # optional last-modified time
# any other producer-defined keys are allowed
---
```

- `type` is the only hard requirement. It is a short, self-descriptive string used for routing, filtering, and presentation. Examples: `Account`, `Territory`, `Sales Play`, `Playbook`, `Metric`, `Reference`, `BigQuery Table`.
- Type values are not registered anywhere. Pick descriptive ones. Consumers must tolerate unknown types (treat as generic).
- Recommended fields in priority order: `title`, `description`, `resource`, `tags`, `timestamp`.
- Producers may add any extra keys. Consumers must preserve unknown keys and must not reject documents for having them.

## Body

Standard markdown. Favor structure (headings, lists, tables, fenced code) over prose; structure helps both human reading and agent retrieval. No required sections. Conventional headings when applicable:

| Heading | Purpose |
|---------|---------|
| `# Schema` | Structured description of an asset's columns or fields |
| `# Examples` | Concrete usage examples, usually fenced code |
| `# Citations` | External sources backing claims (numbered) |

## Cross-linking

Standard markdown links assert an untyped relationship; the surrounding prose says what kind.

- **Absolute (recommended)**: begins with `/`, relative to the bundle root. Example: ``See the customers table at `/tables/customers.md`.`` Stable when files move within a subdirectory.
- **Relative**: Example syntax: neighboring concept at `./other.md` in source bundles; use real repo paths here.

Consumers must tolerate broken links. A link to a not-yet-written concept is valid, not malformed.

## index.md (progressive disclosure)

Optional, at any level including the bundle root. Lets a reader see what exists before opening documents. No frontmatter, except the bundle-root `index.md` may carry a single `okf_version` declaration. Body is sections of links:

```markdown
# Group Heading
* Title 1 (relative-url-1) - short description
* Title 2 (relative-url-2) - short description
```

Entries should reuse the linked concept's `description`. Producers may generate it; consumers may synthesize one when absent.

## log.md (optional)

Flat list of date-grouped entries, newest first. Date headings use ISO `YYYY-MM-DD`. Leading bold word (`**Update**`, `**Creation**`) is convention, not requirement.

```markdown
# Directory Update Log
## 2026-05-22
* **Update**: Added Customer Metrics doc at `/tables/customer-metrics.md`.
```

## Citations

When the body makes externally sourced claims, list them under a numbered `# Citations` heading at the bottom. Links may be absolute URLs, bundle-relative paths, or paths into a `references/` subdirectory that mirrors external material as first-class concepts.

## Conformance

A bundle is conformant with OKF v0.1 if:

1. Every non-reserved `.md` file contains a parseable YAML frontmatter block.
2. Every frontmatter block has a non-empty `type` field.
3. Every `index.md` and `log.md` present follows its structure.

Consumers treat everything else as soft guidance and must not reject a bundle for: missing optional fields, unknown `type` values, unknown extra keys, broken cross-links, or missing `index.md`. This permissive model is intentional; OKF stays useful as bundles grow and get partially generated by agents.

## Versioning

Version is `<major>.<minor>`. Minor bumps add backward-compatible things (new optional fields, new conventional headings). Major bumps may break (rename required fields, change reserved filenames). A bundle may declare its target version with `okf_version: "0.1"` in the bundle-root `index.md` frontmatter (the only place frontmatter is allowed in an index). Consumers that do not understand the declared version attempt best-effort consumption.

## How to apply it (producer checklist)

1. Give every concept `.md` a frontmatter block with at least `type`. Add `title`, `description`, `timestamp` when cheap.
2. Put frontmatter at the very top. A block anywhere else does not count and breaks conformance.
3. Name directory listings `index.md` and histories `log.md`; do not use those names for concepts.
4. Declare `okf_version: "0.1"` in the bundle-root `index.md`.
5. Prefer absolute (`/path.md`) cross-links.
6. Keep bodies structural. Put external sources under `# Citations`.
